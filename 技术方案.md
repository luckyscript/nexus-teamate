# Nexus Teammate 技术方案

## 1. 文档目标

本文档用于基于《产品文档》细化 Nexus Teammate 的技术实现方案，明确系统边界、技术架构、核心服务拆分、关键数据模型、执行链路、非功能设计和分期落地路径。

本文档聚焦“怎么实现”，不重复产品层面的业务价值描述。

## 2. 技术目标

### 2.1 总体目标
构建一个支持多需求源接入、AI 工作流编排、任务自动流转、Agent 执行、经验沉淀与扩展能力接入的企业级协作平台。

### 2.2 技术目标拆解
1. 统一任务模型，承载不同来源需求。
2. 建立事件驱动的工作流引擎，支撑自动化编排。
3. 建立标准化 Agent Runtime，支持配置化执行与治理。
4. 建立技能、胶囊、模板等资产模型，并与 Agent 解耦复用。
5. 建立可扩展的连接器和插件机制。
6. 建立全链路观测、审计、权限与多租户隔离能力。

## 3. 总体架构

### 3.1 架构原则
1. 任务为核心对象，所有流程围绕任务推进。
2. 事件驱动而非页面驱动，确保自动化和异步执行可扩展。
3. 配置优先，减少硬编码流程。
4. 平台内核稳定，外部能力通过扩展接入。
5. AI 调用与业务状态机解耦，避免流程与模型能力耦合过深。

### 3.2 逻辑分层
建议拆分为六层：

1. 接入层
   负责 Web、OpenAPI、Webhook、第三方连接器接入。
2. 应用层
   负责 AI 看板、Agent 管理、自动化配置、模板中心、经验市场等业务接口。
3. 编排层
   负责事件总线、规则引擎、状态机、工作流调度。
4. 执行层
   负责 Agent Runtime、任务执行器、异步队列、工具调用。
5. 资产层
   负责技能、胶囊、模板、知识索引与版本管理。
6. 基础设施层
   负责数据库、缓存、对象存储、检索、日志、监控、鉴权。

### 3.3 推荐部署形态
MVP 阶段建议采用“模块化单体 + 独立异步执行器”的架构，而不是一开始就全面微服务化。

理由：
1. 模块边界清晰但业务仍在快速变化。
2. 规则引擎、任务模型、资产模型前期会高频演进。
3. 模块化单体更利于快速迭代和事务一致性。
4. Agent 执行和连接器同步可以优先做成独立 worker，先解决异步伸缩问题。

建议阶段：
1. 阶段一：模块化单体 + MySQL + Redis + Queue + 独立 Worker
2. 阶段二：按高负载与高变化域拆分服务，如 Agent Runtime、Connector Hub、Analytics
3. 阶段三：插件与扩展市场独立部署

## 5. 核心模块技术方案

## 5.1 AI 看板

### 5.1.1 技术职责
1. 提供任务的看板和列表视图。
2. 展示任务状态、AI 执行状态、人工接管状态。
3. 接收拖拽、编辑、筛选、评论、批量操作等前端行为。

### 5.1.2 后端实现
1. 基于统一 Task 聚合根返回任务数据。
2. 任务状态变更通过状态机服务校验。
3. AI 执行状态不直接写死在任务表，通过任务执行记录聚合计算。
4. 看板查询优先走读模型，避免直接拼接多表影响性能。

### 5.1.3 前端实现建议
1. 看板视图和列表视图共享同一查询接口，仅切换渲染层。
2. 使用服务端分页和过滤，避免前端一次性加载全部任务。
3. 任务详情侧边栏采用增量加载，降低主看板压力。
4. AI 执行状态通过 WebSocket 或 SSE 推送。

## 5.2 智能密度

### 5.2.1 技术职责
1. 汇总任务、Agent、规则、资产的行为数据。
2. 生成团队、项目、时间维度的分析指标。
3. 为管理端提供可配置的统计口径。

### 5.2.2 实现策略
1. 在线链路中只记录事件，不在交易链路中做复杂统计。
2. 通过埋点事件或领域事件进入分析流水。
3. 采用离线聚合表或准实时聚合表产出报表。
4. 指标计算服务与交易服务解耦。

### 5.2.3 数据路径
1. TaskCreated
2. TaskStatusChanged
3. AgentExecutionStarted
4. AgentExecutionSucceeded
5. AgentExecutionFailed
6. AutomationRuleMatched
7. AssetPublished
8. AssetReused

## 5.3 自动化

### 5.3.1 技术职责
1. 管理规则、触发器、条件、动作。
2. 监听任务与 Agent 执行事件。
3. 基于规则决定是否触发流转或动作。

### 5.3.2 规则引擎设计
规则建议采用声明式 DSL，而不是直接在数据库存脚本。

示例结构：

```json
{
  "name": "high_priority_auto_start",
  "trigger": "task.created",
  "priority": 100,
  "conditions": [
    {"field": "priority", "op": "in", "value": ["P0", "P1"]},
    {"field": "fields.spec_ready", "op": "eq", "value": true}
  ],
  "actions": [
    {"type": "bind_agent", "agent_key": "system:risk_assessment"},
    {"type": "transition_status", "to": "in_progress"}
  ]
}
```

### 5.3.3 引擎组成
1. Trigger Adapter
   将领域事件转换为规则输入。
2. Condition Evaluator
   负责条件表达式求值。
3. Action Dispatcher
   负责执行状态迁移、Agent 触发、通知、写入资产等动作。
4. Rule Recorder
   记录规则命中与执行结果。

### 5.3.4 技术约束
1. 规则执行必须幂等。
2. 动作执行需要支持失败补偿。
3. 同一任务并发命中多规则时，需要按优先级和互斥组处理。
4. 状态流转必须经过统一状态机校验，不允许规则直接跳写数据库。

## 5.4 Agent

### 5.4.1 技术职责
1. 管理 Agent 元数据、配置、版本、权限和发布状态。
2. 提供 Agent 执行入口。
3. 管理 Agent 工具调用、上下文加载、结果落库与重试。

### 5.4.2 Agent Runtime 组成
1. Agent Registry
   保存 Agent 定义、版本、输入输出 Schema。
2. Agent Orchestrator
   生成执行计划，准备上下文和依赖资产。
3. Model Gateway
   封装不同模型供应商的调用能力。
4. Tool Runner
   封装检索、搜索、代码执行、外部 API、插件动作等工具。
5. Execution Store
   保存执行状态、输入、输出、日志、成本、错误。
6. Guardrail Layer
   负责权限校验、敏感信息检测、输出结构校验。

### 5.4.3 Agent 执行链路
1. 自动化模块或人工操作触发 Agent。
2. 生成 Execution 记录，状态为 `queued`。
3. Worker 拉取任务并装载 Task 上下文、技能、胶囊、模板。
4. Agent Runtime 解析 Prompt 模板与输入 Schema。
5. 调用模型或工具执行。
6. 对输出做结构校验与策略校验。
7. 结果写入执行记录，并发布执行成功或失败事件。
8. 若配置了后续动作，则回到自动化引擎继续流转。

### 5.4.4 Agent 定义模型
建议定义以下核心字段：
1. `agent_key`
2. `name`
3. `category`
4. `owner_type`
5. `status`
6. `prompt_template`
7. `input_schema`
8. `output_schema`
9. `tool_policy`
10. `asset_bindings`
11. `model_config`
12. `retry_policy`
13. `timeout_seconds`
14. `version`

## 5.5 扩展中心

### 5.5.1 技术职责
1. 管理第三方连接器和插件。
2. 统一接入认证、回调、同步和错误处理。
3. 暴露插件能力给自动化和 Agent 使用。

### 5.5.2 连接器架构
建议采用 Hub 模式：

1. Connector Definition
   描述支持的事件、字段映射、认证方式。
2. Connector Instance
   某租户实际安装后的配置实例。
3. Sync Job
   定时或事件驱动的数据同步任务。
4. Webhook Receiver
   接收第三方回调并转换为内部事件。

### 5.5.3 插件机制
插件建议至少暴露三类能力：
1. Trigger
2. Action
3. Tool

这样自动化和 Agent Runtime 才能共享插件生态，而不是各做一套。

## 5.6 经验市场

### 5.6.1 技术职责
1. 管理技能、胶囊、案例、规则包等资产。
2. 提供上架、审核、版本、检索、绑定和复用统计。
3. 支持资产与 Agent、模板、项目场景关联。

### 5.6.2 存储策略
1. 结构化元数据存 MySQL。
2. 长文本正文存对象存储或正文表。
3. 检索索引存全文索引或向量索引。
4. 关系绑定单独建表，不混入正文结构。

### 5.6.3 检索策略
MVP 先采用“标签过滤 + 关键词搜索 + 人工推荐”。

第二阶段再增加：
1. 语义召回
2. 按任务上下文智能推荐
3. Agent 执行前自动候选注入

## 5.7 模板中心

### 5.7.1 技术职责
1. 管理看板模板、流程模板、Agent 模板等标准配置。
2. 支持一键创建项目或能力配置。
3. 支持模板升级与差异比对。

### 5.7.2 实现建议
模板本质上是“配置快照 + 元数据”，不建议设计成复杂脚本系统。

模板对象结构建议包含：
1. `template_type`
2. `template_name`
3. `scope`
4. `payload`
5. `version`
6. `publish_status`

## 7. 关键流程时序

## 7.1 需求接入到任务创建

```text
Third Party / Webhook
  -> Connector Hub
  -> Source Normalizer
  -> Dedup Service
  -> Task Service(Create Task)
  -> Event Bus(task.created)
  -> Automation Engine
```

### 关键点
1. 接入层只做标准化，不做复杂业务判断。
2. 去重结果需保留人工修正入口。
3. 创建任务后统一发领域事件，后续流程全部异步触发。

## 7.2 Agent 驱动任务流转

```text
Task Event
  -> Automation Engine
  -> Create AgentExecution
  -> Queue
  -> Agent Worker
  -> Model / Tool / Asset Retrieval
  -> Save Result
  -> Publish agent.execution.succeeded
  -> Automation Engine
  -> Task Status Transition / Notify / Human Takeover
```

## 7.3 经验沉淀

```text
Task Closed / Review Completed
  -> Experience Candidate Generator
  -> Human Review
  -> Asset Publish
  -> Search Index / Recommendation Index
```

## 8. 技术选型建议

## 8.1 后端
建议优先使用 `TypeScript + Midwayjs`。

理由：
1. 对管理后台、工作流配置、插件体系支持较好。
2. 和前端共享类型定义，适合配置驱动场景。
3. Agent Runtime、规则引擎、Webhook 接入都适合事件式编程模型。
4. Midwayjs 在 IoC、装饰器、模块化和云原生部署上与本项目契合度较高。

框架建议：
1. `Midwayjs` 作为服务端主框架
2. `TypeORM` 作为 ORM
3. `BullMQ` 或同类队列用于异步执行
4. `MySQL 8.0` 作为主数据库
5. `Redis` 作为缓存、分布式锁和队列底座

## 8.2 前端
建议使用：
1. `React`
2. `Vite + React`
3. `TanStack Query`
4. 看板拖拽库与虚拟列表结合

如果前后端完全分离，优先保持 `React + Vite` 的轻量构建方式。

## 8.3 搜索与知识检索
MVP：
1. MySQL 基础检索能力 + 业务标签过滤
2. 标签和分类过滤

二期：
1. 向量检索引擎
2. 混合召回

## 8.4 对象与文件存储
1. 对象存储保存附件、模板快照、大文本正文
2. CDN 或静态资源网关用于下载加速

## 8.5 实时能力
1. WebSocket 或 SSE 推送 Agent 执行状态
2. 前端兜底轮询

## 9. 多租户与权限设计

### 9.1 多租户模型
建议采用“共享库、逻辑隔离”的多租户方案。

理由：
1. MVP 成本低。
2. 便于统一升级和运营。
3. 绝大多数业务场景可通过 `tenant_id` 强约束实现隔离。

### 9.2 权限层次
1. 租户级
2. 组织级
3. 项目级
4. 资源级
5. 动作级

### 9.3 权限对象
1. 任务可见范围
2. Agent 配置与发布权限
3. 自动化规则编辑权限
4. 模板与经验市场发布权限
5. 插件安装与密钥配置权限
6. 工具调用权限

## 11. 安全设计

### 11.1 核心安全点
1. 第三方密钥加密存储。
2. Agent 调用工具必须经过权限白名单。
3. 插件动作执行需做租户级隔离。
4. 审计日志不可被普通用户篡改。
5. 模型输入输出需支持敏感信息检测。

### 11.2 Agent 安全护栏
1. 输入长度限制
2. 工具调用次数限制
3. 执行超时限制
4. 输出 Schema 校验
5. 高风险动作人工审批

## 12. 一致性与可靠性

### 12.1 一致性策略
1. 主交易链路采用本地事务。
2. 跨模块异步传播采用 Outbox + Event Bus。
3. 异步消费者必须幂等。
4. 状态流转采用版本号或乐观锁控制并发覆盖。

### 12.2 失败补偿
1. Webhook 入站失败可重放。
2. Agent 执行失败支持重试与死信队列。
3. 自动化动作失败记录补偿任务。
4. 连接器同步失败保留断点续传信息。

## 14. 当前最值得先落地的技术任务

如果现在开始进入研发设计，建议先做以下四件事：

1. 先定义领域模型和数据库 Schema
   这是后续看板、自动化、Agent、经验市场共用的基础。
2. 先定义事件模型
   `task.created`、`task.status_changed`、`agent.execution.*`、`asset.published` 这些事件需要尽早固定。
3. 先定义规则 DSL
   自动化是否可扩展，核心取决于 DSL 设计是否稳定。
4. 先定义 Agent Definition Schema
   输入输出、模型配置、工具策略、资产绑定方式必须标准化。

## 15. 结论

Nexus Teammate 在技术上不应被实现为“聊天机器人加几个页面”，而应被实现为一个以任务为内核、以事件为驱动、以 Agent Runtime 为执行中台、以资产层为复用基础、以扩展中心为生态入口的平台系统。

MVP 阶段最关键的是把四条主干打稳：

1. 统一任务模型
2. 自动化规则引擎
3. Agent Runtime
4. 资产模型

这四部分一旦设计稳定，AI 看板、智能密度、扩展中心、经验市场、模板中心都能持续迭代，而不会陷入重复返工。
