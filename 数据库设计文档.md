# Nexus Teammate 数据库设计文档

## 1. 文档目标

本文档基于当前产品文档与技术方案，结合指定技术栈 `MySQL 8.0 + Midwayjs + React`，细化系统数据库设计，明确核心表结构、字段建议、关系约束、索引策略与分阶段落地范围。

## 2. 设计原则

1. 以任务模型为核心，所有执行、规则、资产围绕任务扩展。
2. 固定高频查询字段显式建列，低频扩展字段使用 `JSON`。
3. 交易链路与分析链路分离，统计型数据不挤占主表。
4. 优先满足 MVP 的稳定性和可演进性，不预先过度拆表。
5. 所有核心业务表默认带 `tenant_id`，支持多租户逻辑隔离。
6. 所有状态变更通过应用层控制，不依赖数据库触发器承载业务逻辑。

## 3. 数据库版本与约定

### 3.1 数据库版本
建议使用 `MySQL 8.0.x`。

原因：
1. 支持 `JSON` 类型。
2. 支持函数索引、窗口函数、CTE 等能力。
3. 更适合后续读模型、统计查询和在线演进。

### 3.2 命名规范
1. 表名使用小写下划线，如 `task_comment`。
2. 主键统一使用 `BIGINT UNSIGNED` 或雪花 ID。
3. 时间字段统一使用 `DATETIME(3)`。
4. 布尔字段统一使用 `TINYINT(1)`。
5. 枚举字段优先使用 `VARCHAR`，避免数据库原生 `ENUM` 带来的演进成本。

### 3.3 通用字段约定
除纯关联表外，建议所有核心业务表包含以下字段：

1. `id`
2. `tenant_id`
3. `created_by`
4. `updated_by`
5. `created_at`
6. `updated_at`
7. `deleted_at`

其中：
1. `deleted_at` 为逻辑删除字段，未删除时为 `NULL`
2. 高频表若不做逻辑删除，可不保留 `deleted_at`

## 4. 数据域划分

数据库按业务域划分为以下几组核心实体：

1. 组织与权限域
2. 项目与看板域
3. 任务与协作域
4. 自动化与工作流域
5. Agent 执行域
6. 资产沉淀域
7. 扩展接入域
8. 审计与分析域

## 5.3 任务与协作域

### 5.3.1 task
全系统核心主表。

```text
task
- id BIGINT UNSIGNED PK
- tenant_id BIGINT UNSIGNED
- project_id BIGINT UNSIGNED
- board_id BIGINT UNSIGNED
- source_type VARCHAR(32)
- source_ref VARCHAR(128) NULL
- external_id VARCHAR(128) NULL
- title VARCHAR(255)
- description MEDIUMTEXT NULL
- status_code VARCHAR(64)
- priority VARCHAR(16)
- assignee_id BIGINT UNSIGNED NULL
- reporter_id BIGINT UNSIGNED NULL
- ai_state VARCHAR(32)
- takeover_state VARCHAR(32)
- parent_task_id BIGINT UNSIGNED NULL
- due_at DATETIME(3) NULL
- custom_fields JSON NULL
- extra_json JSON NULL
- version INT UNSIGNED DEFAULT 0
- created_at DATETIME(3)
- updated_at DATETIME(3)
- deleted_at DATETIME(3) NULL
```

索引建议：
1. `idx_tenant_project_status(tenant_id, project_id, status_code)`
2. `idx_tenant_board_status(tenant_id, board_id, status_code)`
3. `idx_tenant_assignee_status(tenant_id, assignee_id, status_code)`
4. `idx_tenant_parent(tenant_id, parent_task_id)`
5. `idx_tenant_source_ref(tenant_id, source_type, source_ref)`
6. `idx_tenant_created_at(tenant_id, created_at)`
7. `idx_tenant_due_at(tenant_id, due_at)`

设计说明：
1. `status_code` 不直接存中文文案，便于状态机和模板配置。
2. `version` 用于乐观锁，避免拖拽和自动流转并发覆盖。
3. `custom_fields` 承载项目级差异字段。
4. `extra_json` 用于低频附加上下文，不污染主字段。

## 6. 关键关系说明

1. 一个 `tenant` 下有多个 `organization`、`user`、`project`
2. 一个 `project` 下有多个 `board`
3. 一个 `board` 下有多个 `board_column`
4. 一个 `task` 属于一个 `project` 和一个 `board`
5. 一个 `task` 可以关联多个 `task_comment`、`task_event`、`agent_execution`
6. 一个 `automation_rule` 可作用于项目级或看板级
7. 一个 `agent_definition` 可以关联多个 `asset_binding`
8. `skill`、`capsule`、`template` 都可以通过 `asset_binding` 挂到 Agent 或项目

## 7. 索引策略

### 7.1 核心原则
1. 优先为租户维度下的列表查询建立联合索引。
2. 优先覆盖状态、负责人、项目、创建时间等高频筛选字段。
3. 避免在大文本列上直接建普通索引。
4. JSON 字段不做默认索引，只对稳定热点字段做生成列索引。

### 7.2 示例
如果 `task.custom_fields` 中有高频字段 `spec_ready`，建议方式为：

1. 新增生成列 `spec_ready_gen`
2. 对 `spec_ready_gen` 建索引
3. 由应用或数据库生成列保持同步

## 8. 分库分表建议

MVP 阶段不建议分库分表。

原因：
1. 当前业务复杂度更高于数据规模复杂度。
2. 任务、规则、执行、资产之间关系较密集，过早拆分会放大事务和查询成本。
3. 单库配合合理索引足以支撑早期版本。

后续仅在以下场景考虑拆分：
1. `agent_execution` 与 `agent_execution_log` 增长过快
2. `task_event` 和 `audit_log` 成为超大流水表
3. 大租户数据量远超平均水平

优先策略：
1. 先冷热分离
2. 先按时间归档
3. 再考虑按租户或时间分表

## 9. Midwayjs 对应实现建议

### 9.1 ORM
建议使用 `TypeORM`。

原因：
1. 与 Midwayjs 集成成熟。
2. 适合模块化实体管理。
3. 对复杂关联与事务处理足够可控。

### 9.2 Repository 分层建议
每个领域模块建议拆为：

1. `entity`
2. `repository`
3. `domain service`
4. `application service`
5. `controller`

例如任务域：

1. `task.entity.ts`
2. `task.repository.ts`
3. `task-domain.service.ts`
4. `task-app.service.ts`
5. `task.controller.ts`

### 9.3 事务建议
1. 创建任务时同步写 `task`、`task_event`、`outbox_event`
2. Agent 执行结果写入时同步写 `agent_execution`、`task_event`、`outbox_event`
3. 不在同一事务中做耗时外部调用
