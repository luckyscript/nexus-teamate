# Nexus Teammate 接口设计文档

## 1. 文档目标

本文档基于当前产品文档、技术方案和数据库设计文档，细化 Nexus Teammate 的接口设计，覆盖前端业务接口、管理端配置接口、Agent Runtime 执行接口、扩展接入接口和实时推送接口。

本文档默认技术栈：
1. 前端：`React + Vite`
2. 服务端：`Midwayjs`
3. 数据库：`MySQL 8.0`

## 2. 接口设计原则

1. 接口以资源和动作结合设计，避免纯 RPC 混乱扩张。
2. 查询接口优先稳定、可分页、可筛选。
3. 写接口优先显式表达业务动作，不让前端拼装底层状态机逻辑。
4. 所有接口默认租户隔离。
5. 所有异步执行动作都要有可追踪的执行 ID。

## 3. 基础约定

## 3.1 Base URL

统一前缀建议：

```text
/api/v1
```

按域划分：

1. `/api/v1/auth`
2. `/api/v1/projects`
3. `/api/v1/boards`
4. `/api/v1/tasks`
5. `/api/v1/automation`
6. `/api/v1/agents`
7. `/api/v1/assets`
8. `/api/v1/connectors`
9. `/api/v1/analytics`
10. `/api/v1/system`

## 3.2 鉴权与租户

建议使用：
1. `Authorization: Bearer <token>`
2. `X-Tenant-Id: <tenantId>`
3. `X-Request-Id: <requestId>`

说明：
1. `tenant_id` 最终以后端鉴权上下文为准，Header 仅用于网关路由或显式切换。
2. 管理端和业务端共用一套认证体系。

## 3.3 通用响应格式

成功响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {},
  "requestId": "req_xxx"
}
```

分页响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "list": [],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 120
    }
  },
  "requestId": "req_xxx"
}
```

失败响应：

```json
{
  "code": "TASK_STATUS_INVALID",
  "message": "invalid task status transition",
  "data": null,
  "requestId": "req_xxx"
}
```

## 3.4 错误码建议

1. `UNAUTHORIZED`
2. `FORBIDDEN`
3. `VALIDATION_ERROR`
4. `RESOURCE_NOT_FOUND`
5. `TASK_STATUS_INVALID`
6. `AUTOMATION_RULE_INVALID`
7. `AGENT_EXECUTION_FAILED`
8. `CONNECTOR_AUTH_FAILED`
9. `ASSET_PUBLISH_DENIED`
10. `INTERNAL_ERROR`

## 3.5 分页与筛选规范

列表接口统一支持：
1. `page`
2. `pageSize`
3. `keyword`
4. `sortBy`
5. `sortOrder`

筛选类参数尽量使用显式 Query 参数，不建议一个大 JSON 串。

### `POST /api/v1/tasks/{taskId}/transition`


1. 不允许前端直接写 `status_code`
2. `version` 用于乐观锁
3. 服务端负责校验状态机与自动化联动
